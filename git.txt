명령어 없이 깃 배우기(소스트리)
1. 왜 버전을 관리해야 하는지
2. 버전은 어떤 과정을 통해 만들어지는지
3. 만들어진 버전은 어떻게 관리할 것인지

1장 준비하기
1.1 깃이 없는 세상
1. 변경 내역을 확인하기 어렵다.
2. 버전을 되돌리기 어렵다.
3. 협력하기 어렵다.

1.2 버전과 버전 관리 이해하기
버전: 유의미한 변화가 결과물로 나온 것 또는 저장된 것
개발에서 버전관리
- 누가 어떻게 변경했는지 변경 내역들을 기억
- 필요하다면 특정 시점의 버전으로 되돌림
- 여러 명이 협업하는 과정에서 코드를 쉽게 나누고 합침

1.3 깃, 소스트리, 깃허브 알아보기
깃: 버전 관리를 도와주는 소프트웨어
소스트리: 사용자 입장에서 더 편리하게 사용하도록 만든 클라이언트 프로그램
깃허브:  원격 저장소 호스팅 서비스
1. 깃으로 버전 관리한 프로젝트를 깃허브에 업로드할 수 있다.
2. 깃허브에 업로드한 프로젝트에 새로운 버전을 추가할 수 있다.
3. 깃허브에 업로드 된 전 세계 개발자들의 프로젝트를 개인 PC에 다운로드 할 수 있다.
4. 깃허브에 업로드된 프로젝트에 코드를 기여하고, 다른 개발자들과 협업할 수도 있다.

2장 깃으로 버전 관리 시작하기
2.1 버전 관리의 큰 그림
- 버전을 만든다 = 특정 순간의 변경 사항(생성, 수정, 삭제)을 기억한다.
- 변경사항을 반드시 버전으로 만들필요는 없음
1. 새로 만들 버전과는 크게 관련없음
2. 중요하지 않음
3. 임시로 변경함
4. 실수로 변경함
등

깃이 관리하는 공간
1. 작업 디렉터리(working directory) 또는 작업 트리(working tree)
- 버전 관리의 대상의 위치하는 공간, 작업 디렉터리에 변경 사항을 만들 수 있다.(눈에 보임)
2. 스테이지(stage) 또는 스테이징 영역(staging area) 또는 인덱스(index)
- 다음 버전이 될 후보가 올라가는 공간(명시적으로 보이지 않음)
3. 저장소(repository)
- 버전이 만들어지고 관리되는 공간(명시적으로 보이지 않음)

add: 스테이지에 추가한다. = staged: 해당파일을 스테이지시킨다.
commit: 새로운 버전을 만든다.(깃으로 만든 버전)

작업 디렉터리의 파일
1. 변경 사항 생성 -- 작업 디렉터리
2. add(staged) -- 스테이지
3. commit -- 저장소

2.2버전 관리 맛보기
버전을 만들기 위해 필요한 것
1. 로컬 저장소
2. 버전 관리할 대상

- 줄 바꿈(Enter)을 하고 저장
- 커밋 메시지(commit message): 버전을 설명하는 메시지, 첫줄에 제목을 작성하고 한 줄 띄고 본문 작성, 개발자에게는 매우 중요한 의사소통 수단

관리 대상(tracked) 파일과 관리 대상이 아닌(untracked)파일
물음표 모양 아이콘: 기존에 변경사항을 추적하지 않았던 새로운 파일, untracked 파일
빼기 모양, 펜 모양 아이콘 : tracked 파일

.gitignore로 무시하기
버전에 포함하지 않을 파일이나 폴더를 자동으로 무시하는 방법
.gitignore파일: 무시할 파일/폴더 목록을 적는 파일
/는 보통 폴더(디렉터리)를 지칭할 때 사용한다.

2.3버전이 쌓여 사용자에게 선보여지기까지
각 커밋에는 고유한 커밋 해시가 있다.
커밋 해시: 각 커밋이 가진 고유한 ID

태그를 붙여 릴리스하기
릴리스(release): 개발한 소프트웨어를 사용자에게 선보이는 것
태그(tag): 유의미한 커밋을 찾기위해 사용하는 것
소프트웨어는 버전을 <연도.월.일> 또는 <숫자.숫자.숫자> 또는 <숫자. 숫자>
vX. Y. Z
X(주(Major)): 기존에 내놓은 버전과 호환되지 않을 정도로 큰 변화가 있을 때 증가
Y(부(Minor)): 새로울 기능을 추가했을 때 증가
Z(수(Patch)): 기존에 내놓은 버전과 문제없이 호환되며 버그를 수정한 정도의 작은 변화가 있을 때 증가

3장 버전 가지고 놀기
3.1 버전 비교하기
1. 직전 버전과 비교하기
- 추가된 글자는 파란색, 삭제된 글자는 붉은색으로 History에서 확인할 수 있다.
2. 버전별 비교하기
- Ctrl을 누를 상태에서 비교할 버전선텍 후 왼쪽 하단 파일명 클릭

3.2 작업 되돌리기
1. 스테이지에 올라가지 않은 파일 되돌리기
마우스 오른쪽 클릭
폐기: 변경사항을 폐기하여 되돌리기
재거: 파일을 제거하기
2. 스테이지에 올라간 파일 되돌리기
- 선택 내용 스테이지에서 내리기 또는 우측의 -를 클릭
3. 커밋 되돌리기
revert: 기존의 버전은 삭제되지 않는다.
마우스 오른쪽 클릭->커밋 되돌리기
reset: 되돌아갈 버전의 시점으로 완전하게 되돌아가는 방식
마우스 오른쪽 클릭->이 커밋까지 현재 브랜치를 초기화
soft reset: 커밋했다는 사실만 되돌림
mixed reset: 스테이지와 커밋을 되돌림
hard reset: 작업 디렉터리 내 변경 사항까지 통째로 되돌림

3.3 스태시로 작업 임시 저장하기
스태시(stash): 임시 저장 기능, 썩 마음에 들지 않지만 버리기는 아까울 때, 다른 중요한 일을 처리해야 할때
변경사항을 추적하는(tracked) 파일 스테이지에 이미 올라와 있거나 한번이라도 커밋한 적이 있는 파일에만 사용할 수 있음

4장 브랜치로 나누어 관리하기
브랜치(branch): 버전을 여러 흐름으로 나누어 관리하는 방법, 버전의 분기
1. 브랜치를 나눈다.
2. 각자의 브랜치에서 작업한다.
3. (필요한 경우) 나눈 브랜치를 합친다.
- 병합과정에서는 같은 코드또는 파일을 다르게 수정한 부분만 샆펴보면 된다.

1. 브랜치 나누기
master 브랜치: 깃이 제공하는 가장 기본적인, 최초의 브랜치
HEAD: 현재 작업 중인 브랜치의 최신 커밋
체크아웃: 특정 브랜치에서 작업할 수 있도록 작업 환경을 바꾸는 것
- 브랜치 이름만 봐도 목적을 알 수 있도록 이름 설정

2. 브랜치 병합하기
병합(merge): 브랜치를 하나로 통합하는 것
빨리 감기 병합: 변화가 없었던 브랜치가 마치 빨리 감기 하듯 업데이트되는 병합 기법(master 브랜치는 어떤 새로운 커밋도 없이 가만히 있음)
빨리 감기 병합이 아닌 병합: 각 브랜치에 없는 커밋이 한 쪽에는 있고 한쪽에는 없는 경우(병합하면 새로운 커밋 생성)

3. 충돌 해결하기
충돌: 병합하려는 두 브랜치가 서로 같은 내용을 다르게 수정한 상황에서 사용자가 어떤 내용을 반영할 것인지 직접 선택해야 함.

<<<<<<< HEAD
master
=======
foo
>>>>>>> foo

<<<<<<<기호와 =======기호 사이의 내용을 선택할지, 
=======기호와 >>>>>>>기호 사이의 내용을 선택할지 고르라는 표기

내것을 이용해 해결 = 현재 체크아웃된 브랜치의 내용을 반영
저장소 것을 사용하여 해결 = 병합하려는 브랜치의 내용을 반영
- 충돌해결후 커밋 메시지 확인하고 커밋해야 해결

4. 브랜치 재배치하기
rebase: 브랜치 재배치, 브랜치가 뻗어나온 기준점을 변경하는 것

5장
5.1 깃허브 소개
깃허브: 개발자의 SNS, 원격 저장소 호스팅 서비스

1.깃허브 구성
1. Recent Repositories: 필자가 최근 이용한 저장소
2. ALL activity: 팔로우하는 사람들의 활동 내역
3. Explore repositories: 깃허브가 추천해주는 저장소

URL: https://github.com/계정이름
STAR = SNS의 좋아요
README.md: 해당 프로젝트의 설치 방법, 사용 방법 등을 담고 있는 파일
Issues: 오류를 제보하거나, 더 다양한 기능을 제안

2. 원격 저장소 호스팅 서비스, 깃허브
로컬 저장소: 내 컴퓨터 속에만 있는 저장소
원격 저장소: 컴퓨터 속에만 있는 저장소가 아닌, 인터넷 세상 어딘가에 있는 다른 컴퓨터 속의 저장소
원격 저장소가 있으면 백업과 협업에 좋음
public: 모두에게 공개된 저장소를 의미하고
private: 모두에게 공개하지 않고 자정한 사용자만 볼 수 있는 저장소
URL: https://github.com/계정이름/저장소 이름

5.2원격 저장소와의 네 가지 상호 작용
1. 클론(clone): 원격 저장소를 복제하기
2. 푸시(push): 원격 저장소에 밀어넣기
3. 패치(fatch): 원격 저장소를 일단 가져만 오기
4. 풀(pull): 원격 저장소를 가져와서 합치기

1. 소스트리와 깃허브 연동하기
SSH(secure Shell): 안전하게 정보를 주고받을 수 있는 통신 방법
키(key) 두 개를 생성(암호, 암호화된 문자열)
1. public key : 공개 키
2. private key : 개인 키
SSH 통신: 암호화된 통신 방법

2. 클론: 원격 저장소 복제하기
클론(clone) : 깃허브상에 존재하는 원격 저장소를 로컬로 복사하여 가져오는 방법
main 브랜치 = master 브랜치
origin : 원격 저장소 경로에 붙은 일종의 별명, 언제든 수정할 수 있다.

3. 푸시: 원격 저장소에 밀어넣기
푸시(push) : 원격 저장소에 로컬 저장소의 변경 사항을 밀어넣는 것

4. 패치: 원격 저장소를 일단 가져만 오기
패치(fatch): 원격 저장소의 변경 사항들을 일단 가져만 오는 방법
- 패치해도 원격 저장소의 내용이 로컬 저장소에 병합되지 않는다.

5. 풀: 원격 저장소를 가져와서 합치기
풀(pull): 패치와 병합을 동시에 하는 방법

5.3 플 리퀘스트: 깃허브로 협업하기
- 원격 저장소에 누구나 푸시할 수 있다면 여러 문제가 발생할 수 있다.
풀 리퀘스트(pull request): 원격 저장소가 내 변경 사항을 풀(pull)하도록 요철(request)하는 방법
1. 기여하려는 저장소를 본일 계정으로 포크하기
2.포크한 저장소를 클론하기
3. 브랜치를 새성 후 생성한 브랜치에서 작업하기
4. 작업한 브랜치를 푸시하기
5. 풀 리퀘스트 보내기

7장 명령어로 버전 관리하기2
7.1 버전 비교하기
git diff : 최근 커밋과 현재 작업 디렉터리의 차이 출력, 커밋한 이후로 작업 디렉터리에서 무엇을 수정했는지 확인하기 위해 사용
git diff --staged/git diff --cached: 최근 커밋과 스테이지 비교하기
git diff <이 커밋 기준으로> <이 커밋이 달라진 점> : 커밋 간 차이를 비교하는 명령
HEAD 뒤에 붙는 ^의 개수 또는 HEAD~ 뒤에 붙는 숫자
git diff <기준이 되는 브랜치> <기준과 비교할 브랜치>

7.2 작업 되돌리기
- 버전을 되돌리는 방법에는 크게 reset(soft reset, mixed reset, hatd reset), revert
1. git reset <되돌아갈 커밋>: 예전 커밋으로 되돌아가기
soft reset : 커밋만 되돌리기
mixed reset : 스테이지까지 되돌리기
hard reset : 작업 디렉터리까지 되돌리기
git reset --soft<되돌아갈 커밋>
git reset/ git reset --mixed <되돌아갈 커밋>
git rest --hard <되돌아갈 커밋>
2. git revert <취소할 커밋> : 해당 커밋을 취소한 새로운 커밋을 추가하는 방식

7.3 작업 임시 저장하기
git stash / git stash -m "메시지" / git stash --message "매시자"
git stash list : 스테시한 작업 목록을 조회하는 명령어
git stash apply <스태시> : 임시 저장된 작업 적용하기
git stash drop <스태시> : 임시 저장된 작업을 지우는 명령어

7.4 브랜치 관리하기
git branch : 현재 브랜치의 목록과 함께 작업 중인 브랜치가 *로 표시된다.
git branch <브랜치> : 브랜치를 만드는 명령어
git checkout <브랜치> : 체크아웃명령어
git checkout -b <브랜치> : 브랜치를 만듦과 동시에 체크아웃하기
git merge <병합할 브랜치> : 브랜치를 병합하는 명령
-충돌이 발생하면 브랜치 이름이(master|MERGING)
git branch -d <브랜치> / git branch --delete <브랜치> : 브랜치를 삭제하는 명령어
git rebase <브랜치> : 브랜치 재배치하기

8장 명령어로 깃허브 다루기
원격 저장소와 상호 작용하는 방법과 깃허브를 다루는 방법
1. 클론(clone) : 원격 저장소를 복제하기
git clone <원격 저장소> : 원격 저장소에 클론하는 명령
git clone <원격 저장소> <클론받을 경로> : 특정 경로에 원격 저장소를 클론받기

2. 리모트(remote) : 원격 저장소를 추가하고, 조회하고, 삭제하기
git remote : 추가된 원격 저장소 목록 조회
git remote add origin <원격 저장소 이름> <원격 저장소> : 원격저장소 추가하기
git remote -v/git remote --erbose : 원격저장소의 이름과 경로 확인
git remote rename <기존 원격 저장소 이름> <바꿀 원격 저장소 이름> : 원격 저장소의 이름을 바꾸는 명령어
git remote remove <원격 저장소 이름> : 원격 저장소를 삭제하는 방법

3. 푸시(push) : 원격 저장소에 밀어넣기
git push <원격 저장소 이름><브랜치 이름> :<원격 저장소 이름>으로 <브랜치이름>을 푸시하는 명령
-u 옵션 : 처음 푸시할 때 한 번만 사용, 이 옵션과 함께 푸시하면 추후 간단히 git push(git pull) 명령만 사용할 수 있음
ex) git push -u origin main

4. 패치(fetch) : 원격 저장소를 일단 가져만 오기
git fetch / git fetch <원격 저장소 이름> : 패치하는 명령
패치는 원격 저장소의 변경 사항을 가져오기만 할뿐이다.
git checkout FETCH_HEAD명령으로도 체크아웃할 수 있다. FETCH_HEAD는 최근에 패치한 브랜치의 최신 커밋을 가리키기 때문 

5. 풀(pull) : 원격 저장소를 가져와서 합치기
git pull / git pull <원격 저장소 이름> : 풀 명령어

6. 깃 명령으로 풀 리퀘스트 보내기
1. 기여하려는 저장소를 본인 계정으로 포크하기
2. 포크한 저장소를 클론하기
3. 브랜치 생성 후 생성한 브랜치에서 작업하기
4. 작업한 브랜치 푸시하기
5. 풒 리퀘스트 보내기

- 계정으로 포크한 저장소와 원격저장소의 커밋을 동일하게 맞춰야 한다.

7. 더 나아가기
